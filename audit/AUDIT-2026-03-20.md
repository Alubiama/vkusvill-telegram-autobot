# System Audit - 2026-03-20

Scope: live runtime, collector, state/db integrity, Telegram, Mini App frontend, hidden-threat review, chaos probes, and a small TRIZ pass.

Overall verdict: the system is operational and the canonical `X:` runtime is healthy, but there are still a few real gaps worth fixing next:
- `today_discounts.json` and pool writes are still non-atomic.
- `COLLECTION_TIMES=` does not fail fast; it silently falls back to `10:00`.
- `runtime_guard` enforces canonical-root checks at runtime, not by throwing on import.
- The repo still has broad `except Exception` swallow points and naive `datetime.now()` uses in collector-side scripts.

## Block 1: Process and runtime
- 1.1 [pass] PID lock works. A second `python -m src.main` from `D:\projects\vkusvill-telegram-autobot` and `C:\Users\Sasha\Documents\vkusvill-telegram-autobot` logged `Refusing to start from non-canonical workspace` and exited instead of starting a duplicate bot.
- 1.2 [pass] Task Scheduler points at the canonical watchdog script. `schtasks /Query /TN "vkusvill-telegram-autobot-watchdog" /XML` showed `X:\vkusvill-telegram-autobot\scripts\ensure-bot-running.ps1` and a 5-minute repetition trigger.
- 1.3 [pass] Chaos kill/recovery worked. Killing the live launcher PID `12368` caused the watchdog to bring back a new launcher+child pair after `117.1s`.
- 1.4 [pass] Old copies refuse to start. Both non-canonical roots logged the refusal message above when launched with their local venv Python.
- 1.5 [warning] `runtime_guard` is not import-time exception based. Importing `describe_runtime_root()` from `D:` and `Documents` returned `('error', ...)` instead of raising; the enforcement is correct, but the contract is status-based, not exception-based.

## Block 2: Collector

## Block 6: Hidden threats
- 6.1 [warning] Broad exception swallowing still exists in several places. Some are intentional safety guards, but there are still `except Exception: pass` paths in `src/bot.py`, `scripts/vkusvill_collect_discounts.py`, and related helpers.
- 6.2 [fail] Atomic writes are missing for runtime JSON artifacts. This is the same issue as Block 3.5, and it affects both the collector and the Mini App snapshot writes.
- 6.3 [warning] Hardcoded paths still exist, though some are intentional. `REGISTRY.md` and the watchdog use the canonical `X:` path by design, but the collector still defaults to the system Chrome profile path if no `--chrome-user-data-dir` is provided.
- 6.4 [pass] Blocking `time.sleep()` is not in the asyncio bot loop. The sleeps found are in supervisor/backoff or one-shot scripts, not inside live Telegram handlers.
- 6.5 [pass] `subprocess.run()` calls in the runtime path have timeouts. `src/providers.py` uses `timeout=self.timeout_sec`, and `scripts/live_system_audit.py` uses a 20-second timeout for PowerShell probes.
- 6.6 [warning] Naive `datetime.now()` calls still exist in collector-side scripts. `scripts/vkusvill_collect_discounts.py` and a few utilities use local naive dates for run-day math, which is okay only if OS time matches the bot's expected timezone.
- 6.7 [pass] Dependencies are pinned. `requirements.txt` is fixed to `python-telegram-bot[job-queue]==21.10`, `python-dotenv==1.0.1`, `playwright==1.54.0`, and `httpx==0.28.1`, and `pip freeze` matched those versions.
- 6.8 [warning] `.gitignore` covers the critical runtime files, but not every generated artifact. Secrets and runtime DBs are ignored, yet `data/backups/` is not explicitly ignored, so backup DBs can still become repo noise.

## Block 7: Chaos testing
- S1 [pass] Phantom PID recovery works. Writing `999999` into a temp `data/bot.pid` and calling `_install_pid_lock()` replaced it with the current PID and logged `replacing stale pid lock`.
- S2 [warning] Corrupted JSON is still a real hazard on the fallback/manual path. `ManualJsonProvider.fetch()` does a direct `json.loads(Path(self.path).read_text(...))` with no recovery; the live bot is mostly insulated by DB-backed reads, but the fallback path can still crash on malformed JSON.
- S3 [warning] Clock-skew risk remains. The collector-side scripts still rely on naive dates in several places, so an OS timezone mismatch can shift day-boundary behavior.
- S4 [warning] Locked Chrome profiles fail with a readable error, but this is still a live dependency. A `system_chrome` run against the default profile failed with `Failed to open Chrome profile...`, while the repo profile worked; operator ergonomics could be better here.
- S5 [not_testable] SQLite lock chaos was not run against the live DB. Holding an exclusive lock on the production state DB would be too disruptive for this audit.
- S6 [warning] Half-write recovery is not fully proven. The bot does have stale/empty fallbacks for its public snapshot path, but the writer is still non-atomic, so a crash mid-write can still leave an invalid artifact.
- S7 [not_testable] GitHub CDN cache-busting was not verified live. I did not publish a new Pages artifact during this audit.
- S8 [warning] Token cascade degrades mostly by warning, which is good, but not fully proven in a forced-missing live probe. The notes and code indicate the intended behavior, but I did not clear the live tokens on the canonical `.env`.
- S9 [warning] Chrome profile bloat is measurable and non-trivial. See Block 2.7: the profile is already a few hundred MB, with `Default` dominating.
- S10 [not_testable] Full reboot recovery was not exercised. Killing every Python and Chrome process on the machine is too destructive for a live audit.

## Block 8: Fundamentals
- F1 [warning] Logging format is structured, but I did not do a separate 2-minute console capture. `src/main.py` configures `%(asctime)s %(levelname)s %(name)s: %(message)s`, and the live audit output shows the bot is emitting timestamped structured logs.
- F2.a [pass] `ORDER_DEADLINE=invalid` fails fast. `load_settings()` raised `ValueError: not enough values to unpack`.
- F2.b [warning] `COLLECTION_TIMES=` does not fail fast. It silently falls back to `[10:00]` instead of rejecting the empty value.
- F2.c [pass] `BOT_TOKEN=` plus a missing `BOT_TOKEN_FILE` fails fast. `load_settings()` raised `ValueError: BOT_TOKEN_FILE does not exist or is empty`.
- F2.d [pass] `TIMEZONE=Invalid/Zone` fails fast. `load_settings()` raised `ZoneInfoNotFoundError`.
- F3 [warning] Repo hygiene is decent for runtime secrets, but generated artifacts still deserve a second look. `.gitignore` already excludes `.env`, `out/`, `data/state.db`, `data/today_discounts.json`, and Chrome profiles, but backup DBs are not covered.
- F4 [pass] Dependencies are pinned and match the environment. `pip freeze` matched `requirements.txt`.
- F5 [not_testable] Concurrent collect + `/app` was not stressed. Running a live collect while intentionally hammering `/app` would be disruptive without a stronger sandbox.
- F6 [pass] Mini App XSS looks contained. Current code uses `textContent` for user-visible item data and only uses `innerHTML` for fixed snippets or stale copy.
- F7 [warning] Timezone handling is mixed. The bot itself uses `self.settings.timezone`, but collector-side day stamps still use naive `datetime.now()`.
- F8 [not_testable] Backup-restore-by-replacement was not performed on the live DB. I validated backup integrity, but I did not swap `data/state.db` out from under the running bot.

## Block 9: TRIZ analysis
- T1 [warning] Autonomy still depends on several fragile externalities: Chrome, Telegram, disk headroom, and Task Scheduler. The layered guards are good, but the dependency surface is still big enough that another hardening pass is justified.
- T2 [pass] The freshness-vs-availability tradeoff is currently handled in the right direction. `force_stale` and the explicit stale notice are a better failure mode than quietly showing yesterday's discounts.
- T3 [warning] Keep the Chrome profile, but prune it surgically. The current profile is large enough that periodic cleanup of cache-like subdirs should stay on the operational checklist.

## Conclusion
The canonical runtime is in good shape, and the audit confirmed that the most recent hardening is actually working in production. The highest-value follow-ups are:
1. Make the collector and snapshot writes atomic.
2. Fix `COLLECTION_TIMES=` validation so empty input fails fast.
3. Decide whether `runtime_guard` should stay status-based or become exception-based at import time.
4. Reduce the broad `except Exception` surface where it is not truly defensive.
